# Prototype 3.2 Plan — Content, Copy, and Usability Refinement

**Project:** Magic Creature Card Maker / Imagination Play Guide  
**Prototype:** P3.2  
**Status:** Planning / Codex handoff ready  
**Purpose:** Execute a focused refinement pass based on P3.1 validation.  
**Core constraint:** One bounded fix at a time.

---

## 1. Prototype Purpose

Prototype 3.2 is a targeted refinement pass.

It is not a new feature expansion.

P3.1 improved the app visually and added a bridge spike, but validation showed several high-impact issues remain:

1. First-time users may not understand that the app creates prompts, not images.
2. Parent-facing question text is still awkward during live play.
3. Text and suggestion content are too hard to edit because they are buried in implementation code.
4. Buttons are not visually distinct enough for non-reading children.
5. A decorative rainbow/background element interferes with readability near the title.
6. Suggestions/examples need improvement because they are core facilitation aids.

P3.2 should improve these issues while preserving current behavior.

---

## 2. Learning Question

> Can targeted content, copy, and usability refinements make the app easier for a normal parent to facilitate during live child co-play without expanding product scope?

Secondary question:

> Can the project structure make question text and suggestion examples easier for the product owner to revise without editing logic-heavy code?

---

## 3. In Scope

- Clarify that the app creates prompts, not images.
- Centralize field/question/suggestion text into an editable structure.
- Rewrite all field questions and examples.
- Improve major button color differentiation.
- Fix rainbow/title readability conflict.
- Add compact parent tips / how-to-use guidance.
- Update prototype docs and top-level durable docs as appropriate.
- Preserve P3.1 creature creation and bridge behavior.

---

## 4. Out of Scope

Do not add:

- Direct AI image generation.
- New bridge output types.
- Saved creature library.
- Printing support.
- PDF export.
- Backend, database, accounts, auth, or persistent storage.
- Framework migration.
- Full story guide.
- Parent DM assistant.
- Major visual redesign.
- New product mode.

## 6. Work Package Sequence

### WP1 — Clarify Prompt-Only Workflow

**Problem:** A first-time parent did not realize the app would not automatically generate an image.

**Goal:** Make it obvious that the app creates a prompt to copy into ChatGPT or another AI image creator.

**Allowed changes:**

- Add short explanatory copy near the intro and/or image prompt output.
- Add wording such as:
  > This app makes a prompt. Copy it, then paste it into ChatGPT or your favorite AI image creator.
- Improve button/help text only as needed.

**Do not change:**

- Prompt-generation logic.
- Bridge output logic.
- Layout beyond minimal copy placement.
- Styling except minor spacing if required.

**Acceptance criteria:**

- A first-time adult can tell that the app produces a prompt, not an image.
- The copy is short and skimmable.
- The app still works exactly as before.

---

### WP2 — Move Text Into Editable Content Structure

**Problem:** Parent-facing question text and examples need iteration but are too difficult to edit if scattered in markup or function logic.

**Goal:** Centralize field/question/suggestion content.

**Preferred implementation:**

- Create `content.js`, loaded before `script.js`, if that stays simple and GitHub Pages compatible.
- Alternative: a top-level `fieldDefinitions` array/object in `script.js`.

**Content structure should include:**

- field id/key
- field title
- short parent-facing question
- suggestion/example text
- placeholder text
- emoji/visual marker if used

**Do not change:**

- Field behavior.
- Prompt output wording.
- Bridge templates.
- Styling.
- App flow.

**Acceptance criteria:**

- Main creature field text is editable in one obvious place.
- Existing inputs and outputs still work.
- No backend, build process, JSON fetch, package dependency, or framework is introduced.
- Works from local file and GitHub Pages.

---

### WP3 — Rewrite Field Questions and Examples

**Problem:** Current wording is directionally useful but awkward during live play.

**Goal:** Rewrite content into a glanceable facilitation format.

**Preferred pattern:**

```text
Field Name — short question
Ideas: silly example, hybrid example, magical example, or anything else.
```

**Design rules:**

- Short.
- Warm.
- Easy to skim.
- Easy to paraphrase.
- Supports parent-led improvisation.
- Examples inspire without constraining.
- Examples support absurd/hybrid creation.
- Avoid “personality” language for young children.

**Suggested field direction:**

| Field | Question | Ideas |
|---|---|---|
| Creature Mix | What is the magic creature made of? | Dragon + pony, raccoon + witch, dinosaur + fairy, or anything else. |
| Magic | What magic does the magic creature have? | Rainbow magic, snack magic, flying, healing, spooky-hunter magic, or anything else. |
| Colors | What colors should the magic creature be? | Rainbow, grey, gold, blue but not orange, or any wild color rule. |
| Home | Where does the magic creature live? | Candy castle, moon cave, underwater palace, sky castle higher than space, or anywhere else. |
| What They Are Like | What is the magic creature like? | Silly, kind, sleepy, sneaky, brave, gentle, or something different. |
| Accessories | What does the magic creature wear or carry? | Books, toys, crown, cape, boots, necklace, backpack, or anything else. |
| Extra Weird Detail | What is one extra silly or magical detail? | Sneezes stars, loves pancakes, has tiny wings, tells jokes, or anything else. |
| Name | What is the magic creature's name? | Name it now, make one up, or decide after the creature feels real. |
| Picture Style | What should the picture feel like? | Cuddly, storybook, candy fantasy, spooky but friendly, dreamy, or bright cartoon. |

**Acceptance criteria:**

- Field text is easier to skim.
- Text lives in editable content structure.
- Adult can glance, ask, and type.
- Suggestions are more useful and playful.
- App behavior remains unchanged.

---

### WP4 — Differentiate Major Buttons With Color

**Problem:** Buttons work for adults, but children who cannot read need clearer visual distinction.

**Goal:** Make major action buttons visually distinct using bright readable colors.

**Major actions to consider:**

- Fill Example
- Make Creature
- Copy Prompt
- Reset
- Bridge output cards
- Bridge copy button

**Design direction:**

- Use distinct bright colors.
- Prefer rainbow-inspired ordering where useful.
- Preserve readability and contrast.
- Do not rely on color alone for state.
- Keep text labels and selected states.

**Acceptance criteria:**

- Major buttons are visually distinct.
- Adult can guide by color.
- Text remains readable.
- App does not become visually chaotic.
- All actions still work.

---

### WP5 — Fix Rainbow / Title Readability Conflict

**Problem:** The first sentence under the title overlaps or visually conflicts with the rainbow/background.

**Goal:** Make header/title/intro text readable on phone and laptop.

**Allowed fixes:**

- Shift text.
- Add stronger header panel background.
- Move decorative rainbow.
- Adjust layering.
- Add padding/margin.

**Acceptance criteria:**

- Header and first intro sentence are readable.
- Decorative background supports mood without interfering.
- No major redesign.

---

### WP6 - Parent Tips / How to Use

**Problem:** First-time adults may need a small facilitation cue before starting.

**Goal:** Add a compact expandable tips section near the top of the app.

**Allowed changes:**

- Add a native `details` / `summary` section after the title/intro and before the creature form.
- Put the tips copy in the editable content structure.
- Add compact mobile-friendly styling that matches the existing playful style.

**Do not change:**

- Prompt-generation logic.
- Bridge output logic.
- App behavior.
- Product scope.

**Acceptance criteria:**

- Tips section is visible near the top.
- Tips section expands and collapses.
- Expanded text is readable and not intrusive.
- Content lives in the editable content structure.
