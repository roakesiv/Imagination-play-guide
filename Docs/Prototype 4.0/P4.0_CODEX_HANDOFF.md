# P4.0 Codex Handoff — Architecture Runway

Use this prompt to launch Prototype 4.0 in Codex.

---

## Codex Prompt

```text
I want to launch Prototype 4.0 for Magic Creature Card Maker.

Read the top-level source-of-truth docs first:
- 00_README.md
- 01_01_PRODUCT_VISION.md
- 01_02_PRODUCT_INSIGHTS_DESIGN_SESSION.md
- 01_03_PRODUCT_ROADMAP.md
- 02_APP_SPEC.md
- 03_UX_SPEC.md
- 04_PROTOTYPE_STARTER.md

Then review the new Prototype 4.0 starter folder:
- Docs/Prototype 4.0/P4.0_README.md
- Docs/Prototype 4.0/P4.0_SCOPE.md
- Docs/Prototype 4.0/P4.0_ARCHITECTURE_PLAN.md
- Docs/Prototype 4.0/P4.0_TEST_PLAN.md

Prototype 4.0 goal:
Create an architecture runway for prompt-based creative outputs. Refactor the prompt/output architecture so Activity Book and Story Guide features can grow without tangling UI behavior, prompt engineering text, and editable UI content.

Important framing:
- V3 is complete.
- V4.0 is not a feature expansion.
- V4.0 should preserve current user-facing behavior.
- V4.1 will handle Coloring + Find-It prompt quality later.
- Do not add Maze, Letter Tracing, Activity Pack, Story Mode, or new output types in V4.0.

Primary architecture objective:
Make this flow explicit and modular:

Creation inputs
→ structured creature data
→ selected output type
→ prompt template
→ generic prompt builder
→ copyable output

Recommended static file/module structure:
- content.js: editable UI copy, field definitions, suggestions, style chips, parent tips, examples
- promptTemplates.js: output prompt templates and prompt-engineering text
- promptBuilder.js: reusable prompt assembly helpers
- script.js: UI rendering, event handling, copy/reset behavior
- styles.css: styling
- index.html: app shell and script loading

Constraints:
- Keep the app static and GitHub Pages compatible.
- Do not add backend, accounts, auth, database, storage, package manager, framework, build step, or direct AI/image API.
- Do not change the user-facing feature set.
- Do not add new output types.
- Preserve P3.2 behavior: creature creation, parent tips, prompt generation, summary/details, bridge outputs, copy behavior, reset behavior.
- Missing fields must still render as "not specified".
- Keep content.js as the editable source for UI text.
- Prompt templates should not directly query DOM fields.
- UI code should gather creature data, then pass the data object to template/builder functions.

Implementation process:
Work one package at a time and stop after each package for review.

Work Package 1:
Assess current code and create/refine P4.0 docs. Summarize proposed files/functions to change. Do not refactor code yet unless I approve.

Work Package 2:
Extract prompt/output templates into promptTemplates.js or an equivalent clear module. Preserve output text and behavior.

Work Package 3:
Create promptBuilder.js or equivalent reusable prompt assembly helpers. Preserve behavior.

Work Package 4:
Make the creature data object/contract explicit through a clear getCreatureData() or equivalent. Ensure templates consume creature data instead of querying the DOM.

Work Package 5:
Create/update prompt output test matrix in P4.0_TEST_PLAN.md or implementation review. Include checks for all existing outputs, missing fields, and output-specific required instructions.

Work Package 6:
Update P4.0_BUILD_LOG.md, P4.0_IMPLEMENTATION_REVIEW.md, and top-level docs with durable architecture changes.

Acceptance criteria:
- App behaves the same from the user perspective.
- App remains static and GitHub Pages compatible.
- No new dependencies or build tools.
- Prompt templates live in a clear dedicated structure.
- Creature data has an explicit object shape / contract.
- Prompt assembly uses reusable helpers or a clearly reusable pattern.
- Existing output types still generate and copy.
- Missing fields still render as "not specified".
- Prompt output test matrix exists.
- Build log and implementation review are updated.
- Durable architecture decisions are promoted to top-level docs.
- V4.1 Coloring + Find-It work can begin without adding more architecture first.

Start with Work Package 1 only.
```
