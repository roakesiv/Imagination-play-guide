# P4.0 Scope — Architecture Runway

**Project:** Magic Creature Card Maker / Imagination Play Guide  
**Prototype:** P4.0  
**Prototype type:** Architecture / NFR runway  
**Status:** Launched / WP1 complete and ready for review

---

## 1. Scope Statement

V4.0 will refactor the app just enough to support near-term Activity Book and Story Guide features without turning the codebase into a tangled collection of prompt strings.

The user-facing app should remain functionally the same.

The internal structure should become more modular.

---

## 2. V4.0 Learning Question

> Can the app’s prompt-output architecture be made modular enough to support multiple activity and story outputs while keeping the static app simple, testable, and GitHub Pages compatible?

Secondary learning question:

> Can the project introduce lightweight architecture and testing concepts without over-engineering the prototype?

---

## 3. Why V4.0 Exists

V3 established a usable baseline app.

The next roadmap stages require many prompt-based outputs:

- Coloring Page
- Find-It Page
- Maze
- Letter Tracing
- Activity Pack
- House Adventure
- Bedtime Story
- Story/DM Guide

If all outputs are added directly into UI/event code, the app will become harder to maintain and test.

V4.0 creates a runway before feature growth accelerates.

---

## 4. In Scope

### Architecture Refactor

- Make creature data shape explicit.
- Separate prompt templates from UI event logic.
- Create or improve a generic prompt/output builder.
- Keep editable copy/content separate from prompt templates and behavior logic.
- Make output template requirements easier to test.

### File / Module Structure

Recommended simple static structure:

```text
index.html
content.js
promptTemplates.js
promptBuilder.js
script.js
styles.css
```

Codex may propose a similar structure if it preserves simplicity and GitHub Pages compatibility.

### Documentation

- Create P4.0 architecture documentation.
- Create P4.0 test plan.
- Update build log and implementation review.
- Promote durable architecture truths into top-level specs.
- Keep `05_ARCHITECTURE_SPEC_DRAFT.md` inside the Prototype 4.0 folder during implementation.
- Decide at V4.0 completion whether to promote a concise top-level `05_ARCHITECTURE_SPEC.md`.

### Testing

- Create a prompt output test matrix.
- Verify existing outputs still work.
- Verify missing fields still use `not specified`.
- Verify all existing output types use current creature data.
- Verify copy/reset behavior still works.

---

## 5. Out of Scope

Do **not** add these in V4.0:

- New activity output types.
- Maze.
- Letter Tracing.
- Activity pack generation.
- Improved Coloring/Find-It prompts beyond necessary migration.
- Story mode.
- Parent DM assistant.
- Direct image generation.
- Printing support.
- PDF export.
- Saved creature library.
- Backend, database, auth, accounts, or persistent storage.
- Framework migration.
- Package manager.
- Build step.
- Complex automated test framework.

---

## 6. Stable Requirements

These requirements must remain true after V4.0:

- App remains static and runs by opening `index.html`.
- App remains GitHub Pages compatible.
- No backend or build tooling is added.
- Creature creation still works.
- Prompt-only workflow remains clear.
- Parent Tips still work.
- `content.js` remains the editable source for UI content.
- Existing bridge outputs still render.
- Copy Prompt still works.
- Bridge Copy still works.
- Reset still works.
- Missing fields render as `not specified`.
- User-facing behavior should not materially change.

---

## 7. Target Architecture Requirements

### 7.1 Explicit Creature Data Object

The app should have a clear creature data shape, conceptually:

```js
creatureData = {
  name: "",
  creatureMix: "",
  magic: "",
  colors: "",
  home: "",
  personality: "",
  accessories: "",
  extraDetail: "",
  pictureStyle: ""
}
```

### 7.2 Prompt Templates

Output prompt definitions should live in a clear template structure.

Examples:

- image prompt / creature profile card
- creature card text
- story
- adventure
- coloring page
- find-it game

Templates should be easy to find and edit.

### 7.3 Prompt Builder

The app should have a generic way to assemble outputs from:

- selected output type
- creature data
- static prompt-engineering instructions
- attribute block
- output-specific requirements
- safety/style instructions

### 7.4 Separation of Responsibilities

| Area | Responsibility |
|---|---|
| `content.js` | UI copy, field definitions, style chips, parent tips, examples |
| `promptTemplates.js` | Prompt/output templates and output-specific prompt engineering |
| `promptBuilder.js` | Generic prompt assembly helpers |
| `script.js` | UI rendering, event wiring, copy/reset behavior |
| `styles.css` | Visual styling and responsive layout |
| `index.html` | App shell and script loading |

If Codex chooses a different structure, it must preserve these boundaries.

## 8. Work Packages

### WP1 — Architecture Docs and Current Code Assessment

Goal:
- Review current code structure.
- Confirm where prompt templates currently live.
- Create/refine P4.0 docs.
- Propose smallest safe refactor.
- Keep the architecture draft in the Prototype 4.0 folder until implementation proves what should become durable.

Acceptance:
- Codex summarizes current architecture.
- Codex confirms exact files/functions to change.
- No code changes yet unless explicitly approved.
- Architecture draft promotion is deferred until V4.0 completion.

### WP2 — Extract Prompt Templates

Goal:
- Move prompt/output templates into `promptTemplates.js` or similarly clear location.
- Preserve existing output text and behavior unless a small migration adjustment is required.

Acceptance:
- Existing output types still exist.
- UI behavior does not change.
- Templates are easier to edit.
- No new output types added.

### WP3 — Create Generic Prompt Builder Helpers

Goal:
- Introduce reusable functions to assemble prompts/outputs from templates and creature data.

Acceptance:
- Creature image prompt and bridge outputs can be generated through a consistent pattern.
- Missing values are handled consistently.
- No user-facing behavior changes.

### WP4 — Clarify Data Model / Contract

Goal:
- Make `getCreatureData()` or equivalent explicit.
- Document data keys used by templates.
- Ensure template functions use the same data contract.

Acceptance:
- Data contract is documented in code comments and P4.0 docs.
- Templates do not directly query DOM.
- UI code gathers data; templates consume data.

### WP5 — Add Prompt Output Test Matrix

Goal:
- Create manual or lightweight test artifacts for existing output types.

Acceptance:
- Test matrix covers all current output types.
- Test matrix checks data use, missing-field behavior, safety, and copy flow.
- No heavy test framework required.

### WP6 — Update Docs and Review

Goal:
- Update P4.0 build log and implementation review.
- Promote durable architecture changes into top-level docs.
- Confirm no user-facing scope expansion occurred.

---

## 9. Acceptance Criteria

V4.0 is complete when:

- [ ] App behaves the same from the user perspective.
- [ ] App remains static and GitHub Pages compatible.
- [ ] No backend, storage, framework, build step, or new dependency is added.
- [ ] Prompt templates live in a clear dedicated structure.
- [ ] Creature data has an explicit object shape / contract.
- [ ] Prompt assembly uses reusable helpers or a clearly reusable pattern.
- [ ] Templates consume creature data instead of querying DOM directly.
- [ ] Existing output types still generate and copy.
- [ ] Missing fields still render as `not specified`.
- [ ] Prompt output test matrix exists.
- [ ] Build log and implementation review are updated.
- [ ] Durable architecture decisions are promoted to top-level docs.
- [ ] V4.1 Coloring + Find-It work can begin without adding more architecture first.

---

## 10. Done-Enough Definition

V4.0 is done enough when adding or editing an output prompt feels like changing a template, not hunting through UI behavior logic.

The app does not need perfect architecture.

It needs enough modularity to support V4.1–V5 without collapsing under prompt-template growth.
