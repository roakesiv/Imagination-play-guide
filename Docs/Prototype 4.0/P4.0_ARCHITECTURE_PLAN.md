# P4.0 Architecture Plan — Prompt-Based Creative Output Runway

**Project:** Magic Creature Card Maker / Imagination Play Guide  
**Prototype:** P4.0  
**Purpose:** Define the architecture runway for Activity Book and Story Guide growth.

---

## 1. Architecture Goal

V4.0 should make the core app flow explicit and modular:

```text
Creation inputs
→ structured creature data
→ selected output type
→ prompt template
→ generic prompt builder
→ copyable output
```

The user-facing behavior should remain the same.

The internal structure should become easier to extend.

---

## 2. Current Architecture Summary

Current app files after V3:

| File | Current Role |
|---|---|
| `index.html` | App shell and markup |
| `content.js` | Editable UI content, field text, suggestions, style chips, parent tips, examples |
| `script.js` | Form rendering, data reading, prompt generation, bridge templates, copy/reset behavior |
| `styles.css` | Visual styling and mobile layout |

Current pain risk:

- `script.js` may become too large if prompt templates, builder logic, and UI behavior all continue growing together.

---

## 3. Target File Roles

Recommended V4.0 static structure:

```text
index.html
content.js
promptTemplates.js
promptBuilder.js
script.js
styles.css
```

### `content.js`

Owns user-facing editable UI content:

- field titles
- field questions
- suggestions
- placeholders
- markers/emojis
- style chips
- parent tips
- Fill Example values
- button labels if useful

Does **not** own output prompt-engineering templates unless the content is purely UI copy.

### `promptTemplates.js`

Owns output prompt templates and prompt-engineering text.

Should include current output types:

- creature profile card image prompt
- creature card text
- story
- adventure
- coloring page
- find-it game

Future output types:

- maze
- letter tracing
- activity pack
- house adventure
- bedtime story

### `promptBuilder.js`

Owns reusable prompt assembly helpers.

Examples:

- normalize missing values
- build attribute block
- assemble template intro + data + constraints + outro
- render template functions
- validate template existence if simple

### `script.js`

Owns UI orchestration:

- render fields from `content.js`
- read form values
- call prompt builder
- render outputs
- wire button events
- copy text
- reset state
- manage selected bridge output

---

## 4. Data Flow

1. `script.js` renders form fields from `content.js`.
2. User fills fields.
3. UI code reads values into `creatureData`.
4. User selects an output type.
5. Prompt builder resolves the template from `promptTemplates.js`.
6. Builder combines template + creature data + requirements.
7. UI displays the result and copy button.

---

## 5. Stable Data Contract

Template functions should consume a creature data object. They should not query the DOM.

Required creature data keys:

| Key | Meaning |
|---|---|
| `name` | Creature name |
| `creatureMix` | Creature type / hybrid mix |
| `magic` | Magic or power |
| `colors` | Color rules |
| `home` | Home / habitat |
| `personality` | What the creature is like |
| `accessories` | Worn or carried items |
| `extraDetail` | Extra silly/magical detail |
| `pictureStyle` | Visual style / vibe |

Missing values should render as:

```text
not specified
```

---

## 6. Template Contract

Each output template should have a clear contract.

Recommended template shape:

```js
{
  id: "coloringPage",
  title: "Coloring Page",
  category: "activity",
  description: "Printable coloring page prompt",
  build: function(creatureData, helpers) {
    return "...";
  }
}
```

Codex may choose a simpler option as long as templates are easy to edit and test.

---

## 7. Common Prompt Requirements

All prompt outputs should:

- preserve silly/specific child ideas
- use current creature data
- handle missing values
- be kid-safe
- avoid scary, violent, horror, or adult content when relevant
- be copyable
- avoid requiring parent prompt engineering
- remain deterministic/template-based for now

---

## 8. Microservice Concept Mapping

Do not build microservices.

Use microservice concepts locally:

| Microservice Concept | Static App Equivalent |
|---|---|
| Service boundary | Module/file boundary |
| API contract | Data object / template contract |
| Service responsibility | Module responsibility |
| Integration test | Prompt output test |
| Shared schema | Creature data shape |
| Loose coupling | Templates do not query DOM |

The learning goal is separation of concerns, not distributed systems.

---

## 9. V4.0 Architecture Decision

Recommended decision:

> V4.0 should introduce `promptTemplates.js` and `promptBuilder.js`, keep `content.js` for UI copy, and keep `script.js` focused on UI orchestration.

If implementation complexity is higher than expected, Codex should pause and propose a smaller refactor.
