# P4.3 Scope — Save Creature + Session Continuity

**Project:** Magic Creature Card Maker / Imagination Play Guide  
**Prototype:** P4.3  
**Status:** Complete / WP5 deferred  
**Prototype type:** Persistence / workflow continuity

---

## 1. Goal

Add lightweight save/restore behavior so the app can survive interrupted play sessions and support multiple creatures.

P4.3 should prevent the parent from losing creature data after reloads, navigation, accidental button presses, or switching between children’s creatures.

---

## 2. Learning Question

> Can simple local persistence reduce parent friction enough to support activity-page generation across interrupted sessions?

Secondary question:

> Can the app support multiple creatures without adding accounts, cloud sync, or backend complexity?

---

## 3. Product Reason

P4.2 live testing showed:

- Creature creation is still fun.
- Activity pages are exciting.
- Kids like the new activity options.
- Generating images is fun briefly, then waiting becomes tiring.
- Kids want to be told when pages are printed and ready.
- The prompt → image → download → print workflow is serious friction.
- Losing creature data is now a significant blocker.
- Accidentally pressing Fill Example can overwrite real creature details.
- Multiple children want their own creatures.

Therefore, the next valuable improvement is not more activity pages. It is session continuity.

---

## 4. In Scope

### WP1 — Discovery / Validation Capture

- Treat the first P4.2 live play test as WP1 discovery.
- Record insights and product implications.
- Promote durable findings into App Spec / Roadmap if appropriate.

### WP2 — Autosave + Restore Current Creature

- Save current form data to browser `localStorage`.
- Restore current form data on page load.
- Preserve current creature across reload/navigation.
- Avoid backend/storage complexity.

### WP3 — Protect Destructive Actions

- Prevent accidental data loss from risky buttons.
- Fill Example should not silently overwrite active creature data.
- Reset should not erase data accidentally.
- Consider marking Fill Example as test/dev helper or moving it away from live flow.

### WP4 — Saved Creature List

- Allow parent to save current creature.
- Allow parent to load saved creature.
- Allow parent to delete saved creature.
- Support multiple creatures, especially one per child.
- Use local browser storage only.

### WP5 — Optional Export / Import Creature Data

Only if simple and useful.

Possible purpose:

- move creature data between phone/laptop
- backup creature data
- recover if local storage is unreliable

This can be deferred if WP2/WP4 are enough.

---

## 5. Out of Scope

Do **not** add in P4.3:

- accounts
- cloud sync
- backend/database
- authentication
- direct image generation
- print automation
- PDF export
- new activity page outputs
- activity pack bundling
- bridge grouping
- story mode
- parent DM assistant
- major visual redesign
- framework/build/package tooling

---

## 6. Recommended Data Model

Use local storage with simple JSON.

Conceptual current creature data:

```js
currentCreature = {
  name: "",
  creatureMix: "",
  magic: "",
  colors: "",
  home: "",
  personality: "",
  accessories: "",
  extraDetail: "",
  pictureStyle: "",
  updatedAt: "ISO timestamp"
}
```

Conceptual saved creature list:

```js
savedCreatures = [
  {
    id: "generated-id",
    name: "Amor",
    data: { ...creatureData },
    createdAt: "ISO timestamp",
    updatedAt: "ISO timestamp"
  }
]
```

Keep storage simple. Do not overbuild.

---

## 7. UX Requirements

### Autosave / Restore

- Parent should not need to think about saving during live play.
- If the page reloads, the last creature should return.
- App should show a small status such as:
  - `Saved locally`
  - `Restored last creature`
  - `Unsaved changes`
- Status should be compact and not distracting.

### Destructive Action Protection

- Fill Example should not overwrite active data without confirmation.
- Reset should not erase data accidentally.
- Destructive actions should be visually and spatially separated from primary actions.
- Confirmations should be simple and understandable.

### Saved Creatures

- Parent should be able to save a named creature.
- Parent should be able to load a creature.
- Parent should be able to delete a creature.
- Saved list should be compact and mobile-friendly.
- Creature name should be used as display label when available.
- If name is missing, use fallback label such as `Unnamed Creature`.

### Multiple Children

- Support at least a few saved creatures.
- Do not require profiles/accounts yet.
- Do not collect personal child names.
- Saved creature names should be fictional creature names, not child names.

---

## 8. Architecture Requirements

- Preserve P4.0 architecture boundaries.
- Use current creature data contract from `05_ARCHITECTURE_SPEC.md`.
- Keep local persistence code small and isolated.
- Consider adding `storage.js` if useful.
- Avoid putting persistence logic directly inside prompt templates.
- Prompt templates should remain unchanged unless needed for save/load behavior.
- App should remain static and GitHub Pages compatible.

Recommended file responsibilities if a new file is added:

| File | Responsibility |
|---|---|
| `storage.js` | localStorage helpers for current creature and saved creature list |
| `script.js` | UI wiring for save/load/delete/restore |
| `content.js` | labels/copy for any new save UI |
| `styles.css` | compact styling for save UI |

### WP5 — Export / Import Creature Data

Optional.

Decision: Deferred.

Reason: Browser-local save solved the immediate creature continuity problem. JSON export/import would add another manual transfer step between phone and laptop, which works against the desired direction of automating everything after the fun creature-creation moment.

Acceptance if implemented:
- User can copy/export creature JSON or text.
- User can import/paste creature JSON or text.
- Invalid import fails safely.
- No backend required.

## 10. Acceptance Criteria

P4.3 is complete when:

- [x] WP1 live play validation is documented.
- [x] Current creature data persists across reloads.
- [x] Last creature can be restored automatically or with clear user action.
- [x] Fill Example no longer silently overwrites real creature data.
- [x] Reset no longer accidentally destroys live work.
- [x] Multiple creatures can be saved, loaded, and deleted locally.
- [x] Creature creation still works.
- [x] Activity outputs still work.
- [x] App remains static and GitHub Pages compatible.
- [x] No accounts, backend, database, package dependencies, build step, or direct AI API added.
- [x] Docs updated.
- [x] Durable learnings promoted into top-level docs.

Verification note:
- Browser smoke testing is not required for every P4.3 work package.
- For narrow static-app changes, static code review, file reference checks, and product-owner review are enough unless the change affects a broad runtime flow or visual layout.
- Browser/mobile checks remain useful for final prototype acceptance or high-risk interaction changes, but they should not block each small WP by default.

---

## 11. Done-Enough Definition

P4.3 is done enough when a parent can:

1. create a creature,
2. accidentally leave/reload,
3. return and recover the creature,
4. save more than one creature,
5. switch between saved creatures,
6. generate activity prompts later.

It does not need cloud sync or perfect print workflow.

P4.3 reached this done-enough definition.
