# P4.3 Codex Handoff — Save Creature + Session Continuity

Use this to launch P4.3 implementation.

---

## Codex Prompt

```text
I want to launch P4.3 for Magic Creature Card Maker.

Read top-level source-of-truth docs first:
- 00_README.md
- 01_03_PRODUCT_ROADMAP.md
- 02_APP_SPEC.md
- 03_UX_SPEC.md
- 05_ARCHITECTURE_SPEC.md

Then review:
- Docs/Prototype 4.3/P4.3_README.md
- Docs/Prototype 4.3/P4.3_SCOPE.md
- Docs/Prototype 4.3/P4.3_DISCOVERY_VALIDATION.md
- Docs/Prototype 4.3/P4.3_BUILD_LOG.md
- Docs/Prototype 4.3/P4.3_IMPLEMENTATION_REVIEW.md

Context:
P4.2 live testing showed that activity pages are fun and useful, but the biggest blocker is now session continuity. Creature data can be lost after reload/navigation, Fill Example can overwrite real details, and multiple children want their own creatures.

Goal:
Add lightweight local save/restore behavior and simple saved creature management.

Work packages:
1. WP2 — Autosave + Restore Current Creature
2. WP3 — Protect Fill Example and Reset
3. WP4 — Saved Creature List
4. WP5 — Optional Export / Import, only if simple and useful
5. WP6 — Closeout Docs

Important:
WP1 is already complete as discovery/validation capture in P4.3_DISCOVERY_VALIDATION.md.

Constraints:
- Use localStorage only.
- Do not add backend, accounts, auth, database, packages, build step, or APIs.
- Do not add direct image generation.
- Do not add print/PDF workflow.
- Do not add new activity page types.
- Preserve current prompt architecture.
- Preserve all existing activity outputs.
- Keep UX mobile-first and simple.
- Do not collect personal child data.

Recommended architecture:
- Consider adding `storage.js` for localStorage helpers.
- Keep persistence helpers small and isolated.
- `script.js` may wire UI events and call storage helpers.
- `content.js` should own any new button labels/help text.
- `styles.css` should own compact save UI styling.

WP2 acceptance:
- Current creature data saves locally.
- Page reload restores current creature.
- Restored creature can generate outputs.
- Existing outputs still work.

WP3 acceptance:
- Fill Example warns before overwriting active data.
- Reset warns before clearing active data.
- Canceling confirmation preserves data.
- Confirmed Fill Example and Reset still work.

WP4 acceptance:
- User can save current creature.
- User can load saved creature.
- User can delete saved creature.
- Multiple saved creatures persist across reload.
- Missing name uses `Unnamed Creature`.
- No personal data required.

Documentation:
- Update P4.3_BUILD_LOG.md.
- Update P4.3_IMPLEMENTATION_REVIEW.md.
- Update P4.3_SCOPE.md status.
- Promote durable requirements into 02_APP_SPEC.md.
- Promote durable UX rules into 03_UX_SPEC.md if needed.
- Promote durable architecture notes into 05_ARCHITECTURE_SPEC.md if storage architecture is added.

Verification:
- Use lightweight WP-level verification unless the change is broad or risky.
- Static review and file-reference checks are enough for narrow static-app work packages.
- Browser/manual review can be reserved for final prototype acceptance or high-risk interaction changes.
- Review reload restore logic.
- Review Fill Example confirmation.
- Review Reset confirmation.
- Review save/load/delete saved creatures.
- Confirm existing activity output generation paths remain unchanged.
- Confirm no new activity types were added.

Start with WP2 only unless I approve continuing.
```
